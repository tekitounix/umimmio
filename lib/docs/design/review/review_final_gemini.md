umiport アーキテクチャ設計（最終版）レビュー報告書対象: umiport アーキテクチャ設計（最終版）日付: 2026-02-08判定: 承認（Sランク評価）1. 総評提示された設計は、Modern C++ (C++20/23) とビルドシステム (xmake) の機能を最大限に活用し、組み込み開発における長年の課題である「ハードウェア依存の分離」と「移植性」を極めて高いレベルで解決している。これまでの反復設計で挙がった懸念点（循環依存、パッケージの凝集度、外部デバイスの扱い、非Newlib環境対応）がすべて論理的かつエレガントに解決されており、実用性と美しさを兼ね備えた「決定版」 と評価できる。2. 評価すべき優れた点A. パッケージ構成の最適化（分離と統合のバランス）以前の設計案では「分離」自体が目的化し、依存関係が複雑化する傾向があったが、最終版では「単独で意味を持つか」という基準により再構築された点が素晴らしい。umiport-boards の統合: umiport パッケージ内に board/ を統合したことで、platform.hh と startup.cc の相互参照（鶏と卵の問題）が完全に解消された。umidevice の独立: MCU依存領域（umiport）とMCU非依存領域（umidevice）を明確に分離したことで、ドライバの再利用性とテスト容易性が最大化された。「知識の帰属先」という観点で完璧な設計である。B. 出力経路の Link-time 注入 (rt::detail::write_bytes)_write() syscall を唯一の経路とするのではなく、シンボル参照による注入へ切り替えた判断は、移植性を飛躍的に高める。WASM / Host 対応: Newlib を持たない環境や、OS の標準出力を使用する環境でも自然に動作する。共存: 組み込み環境では _write() が write_bytes() を呼ぶブリッジとして機能するため、既存の printf も問題なく動作する柔軟性がある。C. xmake ルールによる抽象化add_rules("umiport.board") というインターフェースの導入は、アプリケーション開発者への DX（Developer Experience）を大きく向上させる。「どのソースをビルドすべきか」「どのインクルードパスが必要か」という複雑さを隠蔽し、宣言的に記述できる点は、プロジェクトが大規模化した際に強力な武器となる。3. 実装に向けたアドバイス設計は完成しているが、実装フェーズにおいて以下の点に留意することを推奨する。A. MCUシリーズ内の微細な差異の吸収umiport/mcu/stm32f4/ の粒度で管理されているが、実際には STM32F401 と STM32F429 等でクロックツリーやペリフェラル数に差異がある。対策: memory_map.hh 等の定数定義ヘッダ内においては、xmake から渡される定義（例: STM32F407xx）に基づく #ifdef 分岐を許容する運用ルールを設けること。「ロジックに分岐を持ち込まない」という原則を守りつつ、定数定義での現実的な対応が必要となる。B. Startupコードのオーバーライドumiport.board ルールが自動的に startup.cc をリンクする仕組みは便利だが、ブートローダー開発など独自のスタートアップが必要なケースで競合する可能性がある。対策: xmake ルール側で startup = false のようなオプションを用意するか、ユーザー側で del_files による除外ができる余地を残しておく設計とすること。C. umidevice の型定義の可読性umidevice はトランスポート層（I2C/SPI）をテンプレートで受け取る設計であるため、ボード層（platform.hh）での結合時に型名が長大になりがちである。対策: C++20 の using エイリアスを積極的に活用し、platform.hh の可読性を維持すること。4. 結論本アーキテクチャ設計書は、プロジェクトの「憲法」として十分に機能する完成度にある。「ソースにハード依存を漏らさない」 という当初の目的はこの構成で確実に達成可能である。移行指針も明確であるため、この設計に基づき実装フェーズへ移行することを推奨する。
