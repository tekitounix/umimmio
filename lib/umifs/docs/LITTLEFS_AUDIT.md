# littlefs v2.11 解析評価レポート

対象: `.refs/littlefs/` (littlefs v2.11, ARM Mbed)

## 概要

littlefs は電源遮断耐性を第一に設計された組込み向けファイルシステム。
COW (Copy-on-Write)、CRC検証、メタデータペアによる冗長化を核とし、
ジャーナリング不要でアトミックな更新を実現する。ウェアレベリング機能も内蔵。

## 評価サマリ

| 観点 | 評価 | 備考 |
|------|------|------|
| 電源遮断耐性 | **堅牢** | COW + CRC + gstateにより、最後のコミット境界まで自己整合に復帰する設計 |
| 割り込み安全性 | **条件付き** | `LFS_THREADSAFE`でロック枠組みあり。ISR直接呼出は不可 |
| エラー伝搬 | **良好** | 一貫してエラーコードを伝搬 |
| 破損データ耐性 | **良好** | CRC検証で不整合を自動検出・棄却。一部ASSERTのみの保護あり |
| コード品質 | **良好** | 設計が明確 |

## 1. 電源遮断耐性

### 一貫性と耐久性の区別

littlefs が保証するのは **一貫性（consistency）** であり、**耐久性（durability）** ではない。

- **一貫性**: 電源断後にファイルシステムが自己整合な状態に復帰する。
  破損やクロスリンクのような不可逆な状態には陥らない
- **耐久性**: 最新のデータが保持されることは保証されない。
  最後に確定したコミット境界までの状態に戻る。未syncのデータは失われる

この区別は重要で、littlefs を使えばデータが失われないということではなく、
**ファイルシステムが壊れない** ということである。

### 設計上のメカニズム

littlefs は以下の3層で電源遮断耐性を実現している。

#### 1-1. メタデータペアのCOW

各ディレクトリは2ブロックのメタデータペアとして管理される。更新は常に一方の
ブロックへの追記として行われ、CRC付きコミットマーカーで確定する。

- 電源断時: CRCが不正な最新エントリは無視され、直前の整合状態に自動復帰
- コンパクション時: 新しいブロックに圧縮書き込みし、完了後にペアを切り替え

#### 1-2. CRCによるコミット境界

各メタデータコミットにCRC32が付与される。さらにFCRC（フォワードCRC）により、
コミット後の消去済み領域が期待通りであることも検証する。

- **書き込みが完了していなければ「起きなかったこと」になる**
- ビットフリップや部分書き込みはCRC不一致で自動検出

#### 1-3. gstate（グローバルステート）による分散トランザクション

2つのディレクトリにまたがる操作（rename等）はgstateで管理:

1. `lfs_fs_prepmove` で移動状態を記録
2. 新ディレクトリにエントリをコミット
3. gstateクリア + 旧ディレクトリからエントリ削除

2と3の間で電源断 → 次回マウント時にgstateから移動を検出し `lfs_fs_demove` で
旧エントリを自動削除。一時的にエントリが2箇所に存在するが、
FatFsのクロスリンクと異なり、次回マウントで必ず収束する。

gstate復旧の前提: マウント時にrepairコードが走ること、および関連する
メタデータブロックペアの少なくとも片方が読み取り可能であること。
ブロックデバイス自体が広範に破損している場合はこの限りではない。

### 電源断シナリオ別の動作

| 操作 | 電源断タイミング | 結果 |
|------|------------------|------|
| ファイル書き込み + close | データブロックCOW途中 | CRC不正 → 前回コミット境界まで復帰 (*1) |
| ファイル削除 | メタデータコミット途中 | CRC不正 → 削除されなかったことになる |
| rename（同一ディレクトリ） | コミット途中 | CRC不正 → rename未実行 |
| rename（異なるディレクトリ） | 新エントリ作成後、旧削除前 | gstateで追跡 → マウント時に自動修復 |
| mkfs | フォーマット途中 | スーパーブロック不整合 → マウント失敗（想定通り） |

(*1) 「前回コミット境界」は最後の`lfs_file_sync`または`lfs_file_close`で確定された
状態を指す。closeはflushを含むが、コミット境界の実効性はブロックデバイス層の
`prog`と`sync`が物理書き込み完了を正しく保証していることに依存する（後述 §5）。

## 2. 割り込み安全性・スレッド安全性

### 2-1. LFS_THREADSAFE のロック機構

`LFS_THREADSAFE` 定義時、全公開API（`lfs_file_open`, `lfs_file_write`等）の
入口で `cfg->lock(cfg)` を呼び、出口で `cfg->unlock(cfg)` を呼ぶ。

ロック粒度はAPI呼び出し単位（粗粒度）。内部状態の部分的な保護はない。

### 2-2. lock/unlock コールバック未検証 [High]

`lfs_init` に lock/unlock コールバックのNULLチェックがない。
`LFS_THREADSAFE` 有効でコールバック未設定の場合、最初のAPI呼び出しで
NULLデリファレンスが発生する。

設定ミスで即クラッシュするタイプの問題であり、デバッグが容易なぶん
データ破壊よりはましだが、運用上の事故率は高い。

**推奨:** `LFS_THREADSAFE` 使用時は、init時にlock/unlockポインタの
非NULLを静的に検証するラッパを用意する。

### 2-3. ISRからの呼び出し

**設計上不可。** ブロックデバイスI/O（read/prog/erase）はブロッキング操作であり、
ISRコンテキストでの実行は禁止。ファイル操作はタスクコンテキストで実行し、
OS層のmutex等で排他すること。

## 3. 潜在的な問題点

### 3-1. ASSERT無効ビルドでのスタック保護欠如 [High]

**lfs.c:962**

```c
LFS_ASSERT(sp < LFS_DIR_TRAVERSE_DEPTH);
```

`lfs_dir_traverse` の再帰深度制限がASSERTのみで保護されている。
`LFS_NO_ASSERT` 定義時（プロダクションビルドで一般的）、破損メタデータが
特定条件を満たすとスタックバッファオーバーフローの可能性がある。

再帰深度の上限3はlittlefsの設計意図（traverseの構造上3段で十分）であり、
正常なメタデータでは超えない。実害条件は **破損メタデータによるtraverse暴走** に
限定されるが、スタック破壊は組込みにおいて致命的であり、ASSERTのみの保護は不十分。

`LFS_DIR_TRAVERSE_DEPTH = 3`、`struct lfs_dir_traverse` ≈ 80バイトなので
スタック消費は約160バイト。組込みのスタックサイズ制約下では無視できない。

**推奨:** ASSERTをエラーリターンに置き換える。

```c
if (sp >= LFS_DIR_TRAVERSE_DEPTH) return LFS_ERR_CORRUPT;
```

### 3-2. lfs_npw2 の未定義動作 [Medium]

**lfs_util.h:146**

```c
static inline uint32_t lfs_npw2(uint32_t a) {
    return 32 - __builtin_clz(a-1);
}
```

`a == 1` の場合 `a - 1 = 0` となり、`__builtin_clz(0)` は**未定義動作**
（GCC/Clang仕様）。現在の呼び出しパスでは `a >= 2` が保証されているため
実害はないが、防御的修正の価値はある。

### 3-3. truncate での rcache バッファ再利用 [Medium]

**lfs.c:3772-3785**

```c
lfs_cache_drop(lfs, &lfs->rcache);
res = lfs_file_flushedread(lfs, file, lfs->rcache.buffer, size);
...
memcpy(file->cache.buffer, lfs->rcache.buffer, size);
```

`rcache.buffer` を一時バッファとして再利用している。`lfs_cache_drop` で
rcache を無効化した上でバッファを借用し、読み出し後にfile cacheへコピー。

動作は正しいが、`lfs_file_flushedread` 内部で `rcache` が再有効化される
コードパスとの暗黙的な依存がある。将来の内部変更で壊れる可能性がある。

### 3-4. lfs_file_seek の符号混在 [Low]

**lfs.c:3706-3713**

`off`（int32_t）を `npos`（uint32_t）にキャストする際に符号なしラップアラウンドが
発生しうる。ただし直後の `npos > lfs->file_max` チェック（file_max = 2^31 - 1）
で不正値は確実に検出されるため**実害なし**。

### 3-5. lfs_bd_cmp の hint アンダーフロー [Low]

**lfs.c:140-141**

`hint - i` で符号なしアンダーフローが起きうるが、hintはread-aheadヒントとして
のみ使用されるため機能的影響なし。パフォーマンスへの微影響のみ。

## 4. ウェアレベリング

### 4-1. 動的ウェアレベリング

ブロックアロケータは `lfs->seed`（CRCベースの擬似乱数）から開始位置を決定し、
lookaheadバッファでフリーブロックをスキャンする。O(1)メタデータオーバーヘッドで
合理的な分散を実現。

### 4-2. メタデータのリロケーション

`block_cycles` 設定でメタデータペアの強制リロケーション間隔を制御。
`(block_cycles + 1) | 1` で計算された閾値を超えるとペアを新しいブロックに移動。

### 4-3. 制約と設定上の注意

- 静的ウェアレベリング（使用頻度の低いデータブロックの移動）は行わない
- データブロックの分散は擬似ランダムアロケータに依存
- `lookahead_size` が小さいと特定のブロック範囲に偏る可能性

これらは設計上の既知のトレードオフであり、バグではない。

### 4-4. 寿命に影響する設定

| 設定 | 影響 |
|------|------|
| `block_cycles` | 小さすぎるとメタデータリロケーションが頻発し、erase回数が増加 |
| `lookahead_size` | 小さすぎるとブロック割当が特定範囲に偏り、消耗が集中 |
| `cache_size` | 小さいとprog回数が増加。フラッシュではeraseが最も高コストだが、prog頻度もページ寿命に影響 |

メタデータ更新頻度（ファイルopen/close/syncのたびに発生）とcache設定の組み合わせが
フラッシュ寿命に直結するため、ユースケースに応じた設定調整が必要。

## 5. ブロックデバイス層の前提

littlefs の電源遮断耐性は、最終的に **ブロックデバイス層の品質に支配される**。

### 5-1. prog の前提

- `prog` はページ単位でしか書けない（read-modify-write が必要な場合がある）
- `prog_size` はデバイスの最小書き込み単位と一致しなければならない
- `prog` が返った時点で、データが物理的にプログラムされていることが期待される

### 5-2. sync の前提

- `sync` は物理媒体への書き込み完了を保証しなければならない
- 書き込みキャッシュを持つデバイスでは、`sync` がキャッシュフラッシュを
  含む必要がある
- `sync` が物理完了を待たない実装の場合、電源遮断耐性は成立しない

### 5-3. 設定の整合性

以下の設定が正しくないと、設計どおりの電源遮断耐性が得られない:

| 設定 | 要件 |
|------|------|
| `read_size` | デバイスの最小読み取り単位以上 |
| `prog_size` | デバイスの最小書き込み単位以上 |
| `block_size` | デバイスの消去単位と一致 |
| `cache_size` | `prog_size` の整数倍 |

これらの不整合は `lfs_mount` 時に一部検出されるが、すべてではない。

---

## 6. メモリ使用量

littlefs のRAM使用量は以下の要素で決まる:

- `lfs_t` 構造体: 約 200バイト
- キャッシュバッファ × 2: `cache_size` × 2
- lookaheadバッファ: `lookahead_size`
- ファイルごと: `lfs_file_t` + `cache_size`（ファイルキャッシュ）

スタック使用量は `lfs_dir_traverse` の再帰で最大 `LFS_DIR_TRAVERSE_DEPTH × 80`
バイト程度。組込み環境のタスクスタックサイズを考慮して設定すること。

---

## 7. 総合評価

littlefs v2.11は電源遮断後にファイルシステムが自己整合な状態に復帰することを
設計レベルで実現しているファイルシステム。COW + CRC + gstateの3層防御により、
最後に確定したコミット境界までの状態に戻る。ただし未syncのデータは失われるし、
この保証はブロックデバイス層（prog/sync）の正しい実装に依存する。

Criticalな問題は発見されなかった。注意すべき点:

1. **ASSERT無効ビルドでのスタック保護** [High] — スタック破壊は致命的。
   プロダクション環境では `LFS_ASSERT` をエラーリターンに置き換えることを強く推奨
2. **lock/unlockコールバック未検証** [High] — `LFS_THREADSAFE`使用時は
   init時の静的検証ラッパを用意すべき
3. **lfs_npw2 の UB** [Medium] — 現在のコードパスでは到達しないが、防御的修正が望ましい
4. **ブロックデバイス層の品質** — 電源遮断耐性の実効性はprog/syncの正しさに依存。
   設定の整合性も確認が必要

組込み環境でのファイルシステム選択において、電源遮断後の自己整合性が要件であれば
littlefs を推奨する。ただし「データが失われない」ではなく「ファイルシステムが壊れない」
という保証範囲を正しく理解した上で採用すること。
