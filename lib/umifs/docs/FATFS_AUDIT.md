# FatFs R0.16 解析評価レポート

対象: `.refs/fatfs/source/` (FatFs R0.16, ChaN)

## 概要

FatFs は組込み向け汎用FATファイルシステムモジュール。FAT12/16/32/exFATに対応し、
ANSI C準拠のポータブルな実装。成熟したコードベースだが、FATフォーマット自体の
構造的制約により電源遮断耐性に根本的な限界がある。

## 評価サマリ

| 観点 | 評価 | 備考 |
|------|------|------|
| 電源遮断耐性 | **不十分** | FATフォーマットの構造的限界。ジャーナルなし |
| 割り込み安全性 | **条件付き** | `FS_REENTRANT=1`でmutex枠組みあり。ISR直接呼出は不可 |
| エラー伝搬 | **一部欠落** | 2nd FAT, FSInfoの書き込みエラーが無視される |
| 破損データ耐性 | **不十分** | 循環クラスタチェーンで無限ループの可能性 |
| コード品質 | **良好** | 成熟したコードベース、基本ロジックは堅実 |

---

## 1. 電源遮断耐性

FATファイルシステムはジャーナリングを持たず、複数セクタにまたがる更新をアトミックに
行う手段がない。以下の操作で電源断が発生するとファイルシステムが不整合状態になる。

### 前提: 更新の順序とアトミック単位

FatFsにおける最小アトミック単位は **1セクタのdisk_write** である。
主要な書き込み操作は以下の順序で複数セクタを更新する:

| 操作 | 更新順序 | 非アトミック区間 |
|------|----------|------------------|
| f_write + f_sync | データバッファ → FATチェーン → ディレクトリエントリ → FSInfo | 全区間 |
| f_unlink | ディレクトリエントリ削除 → FATクラスタチェーン解放 | エントリ削除〜チェーン解放 |
| f_rename | 新エントリ作成 → 旧エントリ削除 | 新エントリ作成〜旧エントリ削除 |
| f_truncate | FATチェーン切断 → ディレクトリエントリ更新 | チェーン切断〜エントリ更新 |

いずれの操作も複数セクタ更新の途中で電源断が起きると不整合が発生しうる。

### 1-1. FAT二重化の非アトミック更新 [High]

**ff.c:1064-1069** (`sync_window`)

```c
if (disk_write(fs->pdrv, fs->win, fs->winsect, 1) == RES_OK) {
    fs->wflag = 0;
    if (fs->winsect - fs->fatbase < fs->fsize) {
        if (fs->n_fats == 2) disk_write(fs->pdrv, fs->win, fs->winsect + fs->fsize, 1);
    }
}
```

1st FAT書き込み成功後、2nd FAT書き込み前に電源断 → 2つのFATコピーが不一致。
多くのFAT実装や修復ツールは1st FATを正として扱うため、不一致自体の致命度は
媒体と運用次第。ただし **2nd FATの`disk_write`戻り値が無視されている** 点は
電源断とは独立した品質上の問題であり、修正すべき（後述 §3-1）。

### 1-2. f_unlink — ロストクラスタ [High]

**ff.c:5152-5161**

```c
res = dir_remove(&dj);             // (1) ディレクトリエントリ削除
if (res == FR_OK && dclst != 0) {
    res = remove_chain(&obj, dclst, 0);  // (2) クラスタチェーン解放
}
```

(1)と(2)の間で電源断 → エントリ削除済みだがクラスタは割当済のまま →
**ロストクラスタ（リーク）**。これはデータ破壊ではなく空き領域の減少として
表面化する。定期的なchkdsk相当の修復処理がなければ回復不能で、
繰り返し発生すると最終的にディスクフルに至る。

### 1-3. f_rename — クロスリンク [Critical]

**ff.c:5320-5365**

ソースコード自身がこの問題を認識している:

```c
/* Start of critical section where an interruption can cause a cross-link */
```

新エントリ作成後、旧エントリ削除前に電源断 → 同一クラスタチェーンが
2つのディレクトリエントリから参照される → **クロスリンク**。
一方のファイルを変更すると他方のデータも破壊される。

**本レポートで最もクリティカルな問題。** ロストクラスタと異なり、
クロスリンクは後続の通常操作でデータ破壊が波及する不可逆な状態であり、
修復ツールでもデータの完全な復元は保証されない。

### 1-4. f_sync — データ→メタデータ順序 [Medium]

**ff.c:4228-4282**

f_sync はデータバッファを先にフラッシュし、その後ディレクトリエントリ、FSInfoの
順に更新する。順序自体は正しい（データが先、メタデータが後）。
ディレクトリエントリ更新とFSInfo更新の間で電源断 → FSInfoが不整合になるが、
FSInfoはヒント情報のため次回マウント時のスキャンで回復可能。

### 1-5. f_truncate — サイズ不整合 [Medium]

**ff.c:5063**

```c
fp->obj.objsize = fp->fptr;  // remove_chainの結果に関係なく実行
```

`remove_chain`がエラーでもobjsizeは更新済。次回f_syncでこの不整合なサイズが
永続化される可能性がある。

実害条件: `remove_chain`がエラーを返すのは主に媒体エラーやI/O層の異常時であり、
その時点でファイルシステム全体の整合性も怪しい状況が多い。とはいえ、
I/O層の一時的なエラー（タイムアウト等）でも永続的な不整合を作りうる点は問題。

## 2. 割り込み安全性・リエントランシー

### 2-1. FS_REENTRANT の保護範囲と限界 [High]

`FS_REENTRANT=1` でボリューム単位のmutexが有効になる。ユーザーは
`ff_mutex_create`/`ff_mutex_give`/`ff_mutex_take` を実装する必要がある。

**限界: ロック粒度の不足**

`Files[]`（ファイルロックテーブル）は全ボリューム共有のグローバル配列だが、
保護するmutexはボリューム単位。複数ボリュームへの同時アクセスで`Files[]`が競合する。

**ff.c:1028** (`dec_share`)

```c
Files[i].fs = 0;  /* Free the entry <<<If this memory write is not atomic... */
```

コード自身がこの問題をコメントで認識している。`FF_FS_LOCK` と `FF_VOLUMES > 1`
を同時に使用する場合、`Files[]`テーブル全体を保護するグローバルmutexが別途必要。

### 2-2. ISRからの呼び出し

**設計上、ISRからのファイルシステム操作は不可。** disk I/Oコールバックが
ブロッキング操作であり、mutex取得もISRコンテキストでは禁止。
ファイル操作は必ずタスクコンテキストで実行し、OS層で排他すること。

---

## 3. エラー処理の欠落

以下の問題は電源断とは独立しており、**媒体劣化や一時的なI/Oエラーを検出できず、
静かに破損を進行させる** 点で危険。

### 3-1. 2nd FAT書き込みエラーの無視 [High]

**ff.c:1068**

2nd FATへの`disk_write`の戻り値が完全に無視されている。1st FATは正常に
書き込めても2nd FATの書き込みが媒体エラーで失敗した場合、冗長性が黙って
失われる。以後の電源断で1st FATが破損すると回復手段がなくなる。

### 3-2. FSInfoセクタ書き込みエラーの無視 [High]

**ff.c:1129**

```c
disk_write(fs->pdrv, fs->win, fs->winsect = fs->volbase + 1, 1);
```

戻り値未チェック。`sync_fs`は後続の`CTRL_SYNC`が成功すればFR_OKを返すが、
**CTRL_SYNCの成功は直前のメタデータ書き込み失敗を覆さない。**
呼び出し元はFR_OKを受け取り、FSInfoが正しく永続化されたと誤認する。

### 3-3. exFAT PercInUse書き込みエラーの無視 [Medium]

**ff.c:1138**

exFATのPercInUse更新の`disk_write`戻り値も無視。

### 3-4. CTRL_SYNC とドライバ品質への依存

`f_sync` → `sync_fs` → `disk_ioctl(CTRL_SYNC)` の呼び出しチェーンにおいて、
FatFsはCTRL_SYNCが「物理媒体への書き込み完了を保証する」ことを前提としている。

ドライバ実装がCTRL_SYNCを正しく実装していない場合（例: フラッシュの
プログラム完了を待たずに返る、書き込みキャッシュをフラッシュしない）、
f_syncの保証は成り立たない。§1の電源断議論はすべてCTRL_SYNCが正しく
実装されていることを前提としている。

## 4. 破損データ耐性

### 4-1. 循環クラスタチェーンで無限ループ [High]

**ff.c:1469-1501** (`remove_chain`), **ff.c:4642-4700** (`f_lseek`)

`get_fat`を使ったチェーン走査に循環検出がない。破損FATが循環チェーンを含む場合、
ループが終了しない。`remove_chain`の`do-while (clst < fs->n_fatent)`条件は
各クラスタが有効範囲内なら通過するため、循環チェーンを止められない。

**対策案:** 走査回数カウンタを追加し、`n_fatent`回を超えたら`FR_INT_ERR`を返す。
visitedセットはRAMを消費するため、組込み環境では回数上限が現実的。

### 4-2. exFAT dirbufサイズ検証 [情報]

**ff.c:2161-2183** (`load_xdir`)

`XDIR_NumSec`の最大値は19エントリに制限されている（ff.c:2162）。
`FF_MAX_LFN < 255` の設定では `MAXDIRB` バッファサイズが19エントリ分に
満たないが、ff.c:2183 の memcpy には `i < MAXDIRB(FF_MAX_LFN)` ガードがあり、
バッファサイズを超えるコピーはスキップされる。**オーバーフローは発生しない。**

ただし、`sz_ent > MAXDIRB(FF_MAX_LFN)` の場合でもループは `sz_ent` まで
回り続け、dir_next による余分なディスク読み取りが発生する（機能的影響はない）。

## 5. 整数オーバーフロー

### 5-1. inc_share カウンタ溢れ [Low]

**ff.c:1008**

```c
Files[i].ctr = acc ? 0x100 : Files[i].ctr + 1;
```

読み取りモードで同一ファイルを255回以上開くと`ctr`が0x100になり、
書き込みモードと区別できなくなる。`FF_FS_LOCK`使用時にロジック破綻の可能性。
ただし255同時openは実用上極めて稀であり、バグの種として記録する。

### 5-2. f_write のラップアラウンド検出 [Low]

**ff.c:4118**

exFATモード（FSIZE_t が64ビット）ではオーバーフローチェックがスキップされる。
ファイルサイズ制約により実害は極めて限定的。

## 6. 運用制約と緩和策

FatFsを書き込み用途で使用する場合、以下の運用制約で安全性を高められる。

### 6-1. 禁止すべきAPI

| API | リスク | 代替案 |
|-----|--------|--------|
| `f_rename` | クロスリンク（Critical） | 新ファイル作成 → データコピー → 旧ファイル削除 |
| `f_unlink` | ロストクラスタ（High） | 使用は最小限に。定期的な空き領域チェックを併用 |
| `f_truncate` | サイズ不整合（Medium） | 新ファイルに書き直す方が安全 |

### 6-2. 許容可能な操作パターン

- **create → append → sync**: 最も安全な書き込みパターン。f_syncでデータとメタデータを
  確定させる。電源断時はf_sync時点の状態に戻る（f_sync以降の未sync分は失われる）
- **read-only**: 完全に安全

### 6-3. アプリケーション層での追加対策

- **追記ジャーナル**: 重要データはジャーナルファイルに追記し、チェックポイントで確定
- **二重書き**: 2つのファイルに交互に書き込み、CRC付きヘッダで有効性を判定
- **定期チェック**: ロストクラスタの検出・回収処理を定期実行

---

## 7. 総合評価

FatFs R0.16は基本ロジックが堅実な成熟したライブラリだが、以下の理由から
**電源遮断が起こりうる組込み環境での使用には注意が必要**:

1. **電源遮断耐性は保証されない** — FATフォーマットの構造的制約であり、FatFs固有の
   欠陥ではない。最も危険なのは `f_rename` のクロスリンクで、不可逆なデータ破壊に至る
2. **エラー伝搬に欠落がある** — 2nd FAT、FSInfoの書き込み失敗が黙殺され、
   媒体劣化を検出できずに破損が静かに進行する
3. **破損メディアへの耐性が不足** — 循環チェーンで無限ループ

FatFsの選択が妥当なケース:
- SDカード等のリムーバブルメディアとのPC互換性が必要
- 読み取り専用、またはデータロスを許容できる用途
- §6の運用制約を遵守し、禁止APIを使わない書き込み用途
